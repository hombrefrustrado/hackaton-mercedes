Ficha técnica del proyecto Mercedes Hackathon

Nombre del proyecto: AI FinOps Gateway - Portal de enrutamiento optimizado y gobernanza de costes de modelos LLM.

Objetivo principal: Interceptar y optimizar en tiempo real el consumo de modelos de lenguaje grandes (LLMs) en la empresa, aplicando límites de presupuesto dinámicos y enrutamiento inteligente mediante Machine Learning.

Logros clave alcanzados:

  • Pilar 1 (Visibilidad): Desglose preciso de consumo por modelo y departamento con tarifas oficiales de tokens.
  • Pilar 2 (Gobernanza): Bloqueo de peticiones por superación de presupuesto y predicción ARIMA de gasto.
  • Pilar 3 (Inteligencia): Clasificador KNN con distancia de Gower para enrutar según el prompt.
  • Pilar 4 (Demo y doc): Dashboard web unificado e interactivo y scripts de automatización.
Principios de diseño y arquitectura Evaluación

🌐 Diseño Agnóstico a Proveedores

El gateway funciona de forma uniforme con proveedores locales (Ollama: llama3.2:3b y mistral:7b) y proveedores Cloud (Groq API: llama-3.1-8b-instant), utilizando la especificación de API compatible con OpenAI (/v1/chat/completions). Esto permite conectar instantáneamente servicios alternativos como OpenAI o Anthropic simplemente modificando las variables de entorno de endpoints en el servidor.

🛠️ Separación de Responsabilidades

  • Proxy / Routing (Lógica de tránsito): proxy.py y knn_routing.py interceptan y clasifican peticiones sin acceder a almacenamiento persistente directamente.
  • Almacenamiento (Base de datos): database.py encapsula la persistencia física en SQLite de forma aislada.
  • Reporting / Interfaz (Capa de presentación): Las vistas del portal visualizan costes, forecast y configuraciones comunicándose mediante endpoints REST seguros sin acceder a la base de datos directamente.

🔒 Seguridad: claves de API protegidas

Las claves de API de los proveedores (como GROQ_API_KEY) se configuran y guardan exclusivamente del lado del servidor (mediante variables de entorno en el contenedor Docker) y nunca se exponen a los clientes o consumidores de la API. Los clientes se identifican usando cabeceras de identidad de red (como x-username) y el gateway proxy se encarga de inyectar las credenciales correspondientes antes de redirigir la consulta al proveedor final.

Diagrama de Flujo del Gateway
1. Entrada del Prompt El usuario o script realiza una consulta (ej. al endpoint unificado /api/v1/auto).
2. Clasificación KNN (AI Routing) El proxy extrae el departamento y estima los tokens. La distancia de Gower compara el prompt contra la matriz de entrenamiento y el clasificador KNN decide el mejor modelo.
3. Control de Gobernanza (FinOps) Comprueba en SQLite si el departamento del emisor dispone de saldo presupuestal suficiente. Si excede, bloquea la llamada devolviendo un código 402 Payment Required.
4. Ejecución del Proxy (Streaming SSE) Redirige la consulta al proveedor elegido (Ollama local o Groq API). Soporta streaming en tiempo real usando Eventos de Servidor (Server-Sent Events).
5. Registro de Auditoría y Costes Al completarse la respuesta, calcula el coste de tokens real consumidos, actualiza el saldo del departamento en SQLite y guarda los datos en el log persistente.
Estructura del Proyecto
Módulo / Archivo Propósito en el Gateway
src/app/proxy.py Gestión de URLs de proveedores y enrutamiento streaming.
src/app/database.py Acceso a base de datos SQLite y gestión de triggers.
src/app/utils/forecast.py Algoritmo ARIMA e incremental de gasto predictivo.
src/app/utils/knn_routing.py Motor de inferencia KNN y cálculo de distancia de Gower.
testing/run_benchmarks.py Script de automatización para generar pruebas sintéticas.
testing/kNN.ipynb Cuaderno Jupyter de entrenamiento y validación KNN.
Guía de Pruebas Automatizadas

Para poblar la base de datos de transacciones de forma automática y simular tráfico FinOps, puedes lanzar el script de benchmarks:

python testing/run_benchmarks.py

Este script realizará ráfagas de consultas a los diferentes proveedores utilizando identidades simuladas y completará el histórico del log de auditoría que alimenta el optimizador y los gráficos predictivos del dashboard.
