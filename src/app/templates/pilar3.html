Pilar 3: Enrutamiento Inteligente (AI Router) K-NN Activo

El gateway de control de costes incluye un sistema de enrutamiento inteligente dinámico que intercepta las peticiones enviadas al endpoint /api/v1/auto y selecciona el modelo de IA más apropiado.

🧠 Modelo KNN Implementado (modelo_knn_ej8.joblib)

Hemos integrado exitosamente el clasificador entrenado por el equipo de ciencia de datos. El proxy de Flask procesa en tiempo real las solicitudes entrantes utilizando la siguiente configuración técnica:

  • Algoritmo: KNeighborsClassifier(n_neighbors=3, metric='precomputed')
  • Métrica de Distancia: Distancia de Gower (calculada dinámicamente sobre variables categóricas y numéricas).
  • Variables Predictoras (Features):
    • departamento (Rol/Equipo del usuario: marketing, desarrollo, rrhh).
    • prompt_tokens (Estimado del tamaño de entrada).
    • Procesamiento de lenguaje (NLP) Mediante la técnica de analisis de sentimiento, hemos extraido cuatro métricas a partir de las querys para poder clasificarlas mejor. Este proceso nos permite obtener datos numéricos fácilmente procesables por KNN a partir del texto base. Más info más abajo.
  • Analisis adicional de los datos cimiento:
    • Para etiquetar con su modelo más óptimo la pool de datos a partir de los cuales KNN realiza sus predicciones, tenemos en cuenta el comportamiento de los tres modelos para cada una de las querys. Basandonos en estos datos y en las variables predictoras, clasificamos estos datos de manera más fina.Más info más abajo.
  • Resolución Destino: Enruta de forma fluida a llama3.2:3b, mistral:7b, o llama-3.1-8b-instant.

Próximos Pasos (optimicación a futuro)

En este momento, el análisis NLP de las querys solo funciona en español. Hemos realizado algunos tests y calculamos que traducir (nuestra prueba ha sido desde el inglés) para realizar el análisis. Lo dejamos pendiente para desarrollo futuro, por no añadir complejidad excesiva. Ahora mismo tenemos dos factores limitantes en nuestro proxy, la cantidad de datos clasificados y el algoritmo que empleamos. Si aumentamos el volumen de datos clasificados, utilizando nuestro método de clasificación completo (lanzar la query a los tres modelos, además de hacer el análisis de sentimiento), nuestro algoritmo se volverá más lento (un dato más es una distancia más que calcular, cada vez). Nuestra aproximación actual, aunque buena, podría adaptarse mejor a una mayor cantidad de datos base utilizando el algoritmo ANNOY. Es similar al KNN que usamos actualmente, pero se adapta mucho mejor a volúmenes de datos elevados. También podría usarse una base de datos basada en grafos (cambia la búsqueda de datos a una basada en cercanía, lo que acelera la implementación).

Análisis de Lenguaje Natural y Sentimiento NLP Heurístico

Cada prompt pasa por un análisis lingüístico basado en listas de palabras clave ponderadas para inferir la intención y el tono antes de enrutar la petición. Este análisis opera en cuatro dimensiones independientes:

🚨 Criticidad — Análisis de Urgencia

Detecta señales de alto impacto y urgencia temporal mediante términos como urgente, crítico, producción, vulnerabilidad, incidente, brecha de seguridad. Cada término suma al score de forma ponderada (doble peso para los más graves). Score resultante: 0–10. Esta variable, cuando es alta, beneficia mucho el elegir Mystral.

🔬 Especificación — Complejidad Técnica

Evalúa la densidad de terminología técnica y de dominio: código, sql, docker, kubernetes, api, arquitectura, configurar, depurar. Solicitudes con alta especificación requieren un modelo con mayor capacidad de razonamiento. Beneficia a Groq un score alto

📋 Concretitud — Instrucciones Estructuradas

Identifica si el usuario quiere una respuesta step-by-step mediante marcadores como paso a paso, exactamente, detalla, instrucciones precisas, lista numerada. Implica estructurar bien la respuesta, priorizando modelos más instructivos.

📏 Tamaño de Respuesta — Inferencia de Longitud

Estima si la respuesta esperada es larga (informe completo, guion de 8 minutos, 1000 palabras, exhaustivo) o corta (brevemente, en una frase, resumen). Determina qué modelo optimiza mejor el coste por token.

Métricas de Rendimiento Exhaustivas Composite Score

Para cada query del conjunto base de 60, ejecutamos los 3 modelos en paralelo y recopilamos un conjunto exhaustivo (180 resultados) de métricas reales de ejecución. Con ellas construímos un score comparativo normalizado entre las tres versiones competidoras. El resultado son los 60 datos clasificados con un alto grado de confianza. Hemos realizado este análisis más costoso al conjunto inicial que son el cimiento a partir de la cual K-NN clasifica.

Métrica Cómo se mide Peso proporcional
Idoneidad (Fit) Score NLP normalizado entre los 3 modelos para esa query ≈ 58%
Latencia (s) Tiempo real medido desde la petición hasta el primer token ≈ 12%
Coste ($) Tokens consumidos × tarifa del proveedor (prompt + completion) ≈ 30%

Normalización relativa por query: Los scores de latencia y coste no son absolutos; se calculan comparando los 3 modelos entre sí para esa misma query. El más rápido recibe 10/10, el más lento 0/10. Esto garantiza que siempre gane el mejor modelo disponible, no el "menos malo".

Pesos dinámicos: Los porcentajes anteriores varían en función del análisis NLP. Si se detecta alta criticidad, el peso del fit sube hasta el 65% y el del coste baja al 5%. Para respuestas largas, el coste sube al 50%. El sistema prioriza lo que más importa para cada tipo de tarea.

Manejo de Fallos (Redirección Automática)

El gateway monitoriza la salud de los proveedores. Si el modelo clasificado por el KNN se encuentra caído o inaccesible, el tráfico se redirigirá automáticamente a un nodo secundario de respaldo.

Esquema de Respaldo:

Mistral 7B (Caído) ➔ Llama Groq (Respaldo)
